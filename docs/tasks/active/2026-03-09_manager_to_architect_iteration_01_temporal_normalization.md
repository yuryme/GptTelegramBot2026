# Iteration 1 Temporal Normalization Task

- Date: `2026-03-09`
- From: `manager`
- To: `architect`
- Status: `approved`
- Topic: `iteration_01_temporal_normalization`
- Goal: `Выделить единый слой нормализации даты и времени и убрать временные костыли из нескольких слоев системы`

## Context

Сейчас интерпретация русскоязычных фраз о дате и времени распределена между промптом LLM, постобработкой ответа LLM и финальным вычислением `run_at`. Из-за этого логика размазана по нескольким слоям, плохо расширяется и провоцирует накопление локальных костылей.

## Decision / Task

Нужно выполнить первую архитектурную итерацию: создать единый модуль `temporal_normalizer`, который станет единственной точкой нормализации временной информации перед исполнением команды.

## Scope

- Создать отдельный модуль нормализации, например `app/services/temporal_normalizer.py`.
- Перенести в него текущие временные эвристики из `LLMService` и схем команд.
- Передавать в normalizer исходный текст пользователя, распарсенную команду, текущее локальное время и timezone приложения.
- Возвращать из normalizer нормализованную команду того же доменного типа, пригодную для дальнейшего исполнения.
- Упростить `LLMService`: оставить в нем только вызов модели, получение JSON, валидацию схемы и recovery невалидного JSON.
- Убрать из `LLMService` самостоятельную смысловую правку даты и времени.
- Упростить `resolve_default_run_at()`, оставив там только техническое преобразование уже нормализованной структуры в `datetime`.
- Централизованно поддержать минимум такие сценарии:
- `сегодня`
- `завтра`
- `послезавтра`
- конкретная дата вида `2026-03-10`
- дата без времени
- дата с точным временем
- время в форматах `10:30`, `10.30`, `10-30`
- извлечение времени из `run_at`, если LLM вернула абсолютную дату, но в тексте есть относительный день
- корректную работу с `day_reference = today/tomorrow/day_after_tomorrow/weekday/specific_date`
- корректную поддержку `explicit_time_provided`
- Добавить table-driven тесты на эталонные русские фразы для single-reminder сценариев.
- Сохранить текущую внешнюю совместимость пользовательского сценария и JSON-команд.

## Out Of Scope

- Полная переработка JSON-контракта команд.
- Полный redesign NLP-пайплайна.
- Полная переработка цикличности и графика показа.
- Изменение внешнего пользовательского интерфейса Telegram-бота.
- Переписывание проекта с нуля.

## Acceptance

- В проекте появился отдельный `temporal_normalizer`.
- Временные эвристики больше не размазаны между промптом, `LLMService` и схемами.
- `LLMService` не содержит самостоятельной смысловой логики правки даты и времени.
- `resolve_default_run_at()` больше не является основным местом интерпретации русских временных конструкций.
- Текущие рабочие сценарии не деградировали.
- Добавлены и проходят тесты минимум на такие фразы:
- `Сегодня в 12:15 напомни позвонить`
- `Завтра в 10 купить молоко`
- `Послезавтра в 18:30 встреча`
- `10 марта в 9 поздравить маму`
- `В среду в 14 созвон`
- `Сегодня напомни отправить отчет`
- `Завтра напомни купить билеты`
- Все существующие релевантные тесты, связанные с `LLMService`, временной нормализацией и созданием напоминаний, адаптированы и проходят.
- Обновлены `README.md` и `PROJECT_PLAN.md` с описанием нового слоя и статуса итерации.

## Notes

- На этой итерации не менять внешний контракт команд без отдельного согласования.
- Если часть текущей логики требуется как fallback, она должна использоваться только после normalizer и быть четко отделена от смыслового разбора.
- Результат итерации должен подготовить систему к следующему этапу: введению промежуточного semantic/draft слоя между LLM и финальным JSON.
